# 多 Agent 协作架构讨论记录

> 讨论时间：2026-02-07
> 状态：方案讨论阶段，尚未决定实施

## 背景

当前项目是一个飞书机器人，通过飞书 WebSocket 长连接接收消息，调用 AI SDK 处理请求。在实际使用中发现：

1. **群聊 @ 过滤问题**：两个机器人拉到同一个群里，@任意一个，两个都会回复。原因是代码只检查了 `mentions` 是否非空，没有检查是否 @的是自己。（已修复，通过 `/open-apis/bot/v3/info/` 获取机器人自身 `open_id` 进行过滤）
2. **飞书平台限制**：机器人发的消息不会触发其他机器人的 `im.message.receive_v1` 事件，机器人之间无法通过飞书消息互相通信。这是平台层面防止死循环的设计。

由此引出需求：**能否让多个机器人在群里协作，各自发挥不同 AI 能力？**

## 方案选型

### 方案一：单机器人，内部多 Agent（未采用）

一个飞书机器人作为入口，内部同时调用多个 AI SDK（Claude、Codex、Gemini 等）。

- 优点：简单，无额外通信机制
- 缺点：扩展性差，所有能力耦合在一个服务里；群里只有一个机器人身份，用户看不到"团队协作"的效果

### 方案二：多机器人 + 后端通信通道（采用）

多个独立的飞书机器人，各自有不同的 AI 能力，通过后端 HTTP 通道通信协作。

- 优点：可插拔，新增能力只需部署新机器人；用户在群里能看到多个机器人各自发言，体验像真人团队
- 缺点：需要额外的通信机制和任务管理

## 架构总览

```
飞书群
├── 用户
├── 统筹者机器人 A（固定，Claude）
├── 子机器人 B（Codex）
├── 子机器人 C（Gemini / 搜索 / ...）
└── ...

后端通信层（HTTP + SSE）
  统筹者 A ⟷ 子机器人 B
  统筹者 A ⟷ 子机器人 C
  （星型拓扑，子机器人之间不直接通信）
```

## 核心设计决策

### 1. 统筹者角色

- **固定一个统筹者机器人**，始终监听群消息，负责任务分析、拆分、分发、汇总
- 统筹者本身具备 AI 能力（如 Claude），简单任务自己直接处理，不分发
- 复杂任务（单个机器人 2 步内做不完的）才触发分发流程

### 2. 通信拓扑

- **星型拓扑**：子机器人只能向统筹者汇报，不允许平级直接通信
- 如果子机器人 B 的结果需要交给子机器人 C 使用，路径为：B → 统筹者 → C
- 理由：简单可控，统筹者掌握全局状态

### 3. 群聊发言规则

- 子机器人只要参与了任务，就要在群里发言（让用户看到进展）
- 协商拆分过程走底层 HTTP 通道，但**同时在群里发展示性消息**，给用户提供"情绪价值"
- 后期可优化为统筹者发一条"我们私下沟通一下"，协商过程不在群里展示

### 4. 内部通信协议

- **任务分发**：统筹者通过 HTTP 调用子机器人的 `/task` 接口
- **进度汇报**：子机器人通过 SSE 向统筹者实时推送进度
- **最终结果**：子机器人完成后通过 HTTP 回调提交结果
- 统筹者需要能同时处理多个子机器人的 SSE 流

### 5. 能力注册

每个子机器人在代码中定义自己的能力，暴露 `GET /manifest` 接口：

```json
{
  "name": "codex-bot",
  "capabilities": ["code_generation", "code_review", "debugging"],
  "description": "擅长代码生成和审查",
  "endpoint": "http://bot-b:3000"
}
```

统筹者的配置通过 Docker 环境变量指定子机器人地址列表，启动时逐个拉取 manifest。统筹者将能力描述提供给 AI，由 AI 决定任务分发策略。

### 6. 汇总与结果呈现

- 子机器人各自在群里发执行进展
- 同时通过底层通道向统筹者汇报结果
- 统筹者收齐所有子任务结果后，在群里 @用户 发最终总结

## 任务生命周期

### 完整流程

```
阶段一：接收 & 判断
  用户 @统筹者 → 统筹者 AI 分析意图
  ├── 简单任务 → 统筹者自己答，结束
  └── 复杂任务 → 进入阶段二

阶段二：协商拆分
  统筹者通过 HTTP 把需求发给相关子机器人
  子机器人各自回复"我能做什么部分"
  统筹者 AI 综合后确定任务分配方案
  统筹者在群里发任务分配消息（展示用）

阶段三：执行
  子机器人并行/串行执行
  ├── 各自在群里发进展消息（对用户可见）
  └── 同时通过 SSE 向统筹者汇报进度
  （串行场景：前一个子任务的结果作为后一个的输入，由统筹者中转）

阶段四：汇总
  统筹者收齐所有结果
  在群里 @用户 发最终总结
```

### 任务状态机

```
         用户发消息
             │
             ▼
         ┌─────────┐    队列中有任务时自动取出
    ┌───▶│ pending  │◀────────────────────────
    │    └────┬─────┘
    │         │ 开始处理
    │         ▼
    │    ┌─────────┐   用户说"停一下"   ┌─────────┐
    │    │ running  │ ────────────────▶ │ paused  │
    │    └────┬─────┘                   └────┬────┘
    │         │                              │ 用户补充背景
    │         │                              │ 统筹者更新上下文
    │         │                              │ 重新规划，回到 running
    │         │         ┌────────────────────┘
    │         │         ▼
    │         │    ┌─────────┐
    │         │    │ running  │（带更新后的上下文）
    │         │    └────┬─────┘
    │         │         │
    │         ├─────────┘
    │         ▼
    │    ┌─────────┐
    │    │  done    │ → 统筹者汇总 → 检查队列
    │    └─────────┘
    │
    │   新消息到达时正在 running/paused → 入队列
    │         │
    │    ┌────┴────┐
    └────│ queued  │
         └─────────┘
```

### 暂停机制

- 用户可以主动说"停一下"来暂停当前任务
- 统筹者通知子机器人暂停（HTTP 调用），子机器人停止当前 AI 调用，保留已有结果
- 用户补充上下文后，统筹者带着新上下文**重新规划**（不是续传，而是重新来一轮）
- 已完成的子任务结果作为上下文带入新一轮规划

### 消息队列

- 每个群一个 FIFO 队列
- 用户在任务执行期间发新消息 → 入队，回复"收到，前面还有 N 条在排队"
- 当前任务完成后自动从队列取出下一条处理
- 一个群同时只处理一个任务

## 关键数据结构（概念设计）

### TaskManager

```typescript
interface Task {
  taskId: string;
  chatId: string;          // 来源群
  userId: string;          // 发起用户
  userMessage: string;     // 原始消息
  status: 'planning' | 'executing' | 'paused' | 'summarizing' | 'done';
  context: string;         // 累积的上下文（含用户补充的背景）

interface SubTask {
  subTaskId: string;
  botName: string;
  endpoint: string;
  status: 'pending' | 'running' | 'done' | 'failed';
  progress?: string;       // SSE 汇报的进度
  result?: string;         // 最终结果
}
```

## 待解决的疑点

### 1. 统筹者如何准确拆分任务？

统筹者需要通过 AI 理解用户意图并拆分成子任务，这本身就是一个难题：
- AI 可能误判任务复杂度，简单任务被过度拆分，或复杂任务拆分不合理
- 子机器人的 capability 描述是否足够精确，让 AI 做出正确的路由决策？
- 如果没有任何子机器人能处理某个子任务，统筹者应该怎么办？自己兜底？还是告知用户？

### 2. 协商阶段的深度问题

讨论中确定了"机器人一起协商拆分任务"，但协商到什么程度？
- 子机器人只回复"我能/不能做"，还是要给出具体方案？
- 如果两个子机器人都声称能做同一个子任务，统筹者怎么选？
- 协商本身也消耗 AI tokens，对于简单任务是否跳过协商直接分发？

### 3. 子机器人在群里发言的内容控制

确定了"子机器人参与就要在群里发言"，但需要考虑：
- 子机器人发的是过程信息还是结果信息？过程信息（如"正在搜索..."）和最终结果都发会很冗长
- 多个子机器人同时发消息，群里消息顺序可能混乱，用户阅读体验差
- 子机器人的发言格式需要统一吗？还是各自风格？

### 4. 暂停后重新规划的上下文问题

用户暂停补充背景后重新规划，已完成的子任务结果作为上下文带入。但：
- 已完成的子任务结果可能很长，全部作为上下文会消耗大量 tokens
- 新的背景信息可能导致之前的子任务结果失效，需要重做，但怎么判断哪些需要重做？
- 暂停时正在执行的子机器人，其中间状态如何处理？直接丢弃还是尝试保留？

### 5. 统筹者的单点问题

统筹者是固定的唯一入口：
- 统筹者挂了，整个系统不可用，子机器人即使健康也无法工作
- 统筹者重启后，内存中的 Task 状态、消息队列全部丢失
- 是否需要持久化任务状态（如 Redis）？还是接受重启后丢失？

### 6. 飞书应用权限与身份

每个子机器人都是一个独立的飞书应用：
- 需要各自申请飞书应用、配置权限、拉进群——运维成本随机器人数量线性增长
- 每个飞书应用有独立的 API 调用配额，需要分别管理
- 子机器人是否需要监听群消息？按当前设计只需要在群里发消息，不需要 WebSocket 长连接，只用 API 发消息即可

## 实施难点

### 1. 统筹者的并发模型

统筹者需要同时处理：
- 飞书群消息（用户新请求、暂停指令）
- 多个子机器人的 SSE 进度流
- 自身的 AI 调用（任务拆分、汇总）
- 消息队列的调度

Node.js 单线程模型下，这些都是异步 I/O 可以处理，但状态管理复杂度高。一个 Task 对象会被多个异步流程并发修改，需要仔细处理竞态条件。

### 2. 子机器人的通用框架

每个子机器人需要实现一套标准接口（`/manifest`、`/task`、`/pause`、SSE 推送），如果每个都从头写很浪费。需要抽象出一个子机器人的基础框架/SDK，新增机器人只需要实现业务逻辑。

### 3. 多轮分发的编排

有依赖的子任务需要多轮分发（如先搜索再总结）。统筹者需要理解子任务之间的依赖关系并按顺序编排。这个编排逻辑本身可以让 AI 来做，但会增加 AI 调用轮次和成本。

### 4. 错误处理与降级

- 子机器人超时或失败后，统筹者是重试、换一个机器人、还是跳过这个子任务？
- 如果关键子任务失败导致后续任务无法进行，怎么向用户解释？
- 网络抖动导致 SSE 断开，如何重连和恢复进度？

### 5. 测试与调试

- 多机器人协作场景难以本地复现，需要多个飞书应用同时运行
- 问题排查需要关联多个服务的日志，统一的 traceId/taskId 贯穿所有日志是必须的
- 协商和分发过程的 mock 测试方案需要提前设计

## 未来风险

### 1. 成本风险

- **叠加效应**：每条用户消息 = 统筹者 AI 调用（分析+拆分） + N 个子机器人 AI 调用 + 统筹者 AI 调用（汇总），成本是单机器人的 N+2 倍
- **协商开销**：协商阶段也要消耗 tokens，实际成本可能更高
- **无效调用**：AI 误判导致不必要的分发，浪费子机器人的 AI 调用费用

### 2. 用户体验风险

- **响应变慢**：多了协商、分发、等待、汇总环节，端到端延迟显著增加
- **消息轰炸**：多个机器人在群里发消息，对于只是想问个简单问题的用户来说很吵
- **理解成本**：用户需要学习"哪些情况会触发多机器人协作"，心智模型复杂

### 3. 运维风险

- **部署复杂度**：从一个 Docker 容器变成 N+1 个，加上通信配置、飞书应用管理
- **版本协调**：统筹者和子机器人的接口需要保持兼容，升级时要考虑灰度
- **监控告警**：需要监控每个机器人的健康状态、AI API 配额余量、飞书 API 调用频率

### 4. 规模风险

- **飞书限制**：一个群能拉多少个机器人？飞书可能有上限
- **消息频率**：多个机器人同时发消息，飞书的频率限制（失败重试可以解决，但会引入延迟）
- **群消息刷屏**：机器人数量多了，群消息会被机器人占满，影响人与人的正常沟通

### 5. 安全风险

- 子机器人的 HTTP 接口如果暴露在公网，需要鉴权机制（如 shared secret / mTLS）
- 统筹者转发的上下文可能包含敏感信息，子机器人之间的信息隔离需要考虑
- 子机器人被恶意替换后可能返回错误结果，统筹者是否需要验证结果可信度？

## 与现有代码的关系

当前项目已有的基础：
- `provider.ts`：已有 Claude / Codex 双 Provider 路由机制
- `feishu.ts`：已有会话管理、并发控制、消息去重
- `types.ts`：已有统一的 `ClaudeEvent` 事件类型

如果实施此方案，需要新增：
- `task-manager.ts`：任务状态管理、队列
- `bot-registry.ts`：子机器人注册发现
- `coordinator.ts`：统筹逻辑（任务拆分、分发、汇总）
- 子机器人侧：暴露 `/manifest`、`/task`、`/pause` 等 HTTP 接口 + SSE 进度推送
